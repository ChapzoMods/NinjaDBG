# NinjaDBG v1.2.0 — Windows Port + x64dbg-like Features Plan

**Author:** Plan subagent
**Scope:** Add a Windows .exe + .dll build path (MinGW cross-compile from Linux) and bring NinjaDBG's CLI closer to x64dbg's feature set.

---

## 0. Current State (findings from the v1.1.3 codebase)

| Area | File(s) | Status |
|---|---|---|
| Debugger core (ptrace) | `include/DebuggerCore.h`, `src/DebuggerCore.cpp` (839 LoC) | Linux-only. Hard-depends on `<sys/ptrace.h>`, `<sys/user.h>`, `<sys/wait.h>`, `<sys/uio.h>`, `<sys/mman.h>`, `process_vm_readv`, `PTRACE_*` constants. |
| Public API | `DebuggerCore.h` | Already cleanly factored: `attach`, `attachByLaunch`, `cont`, `step`, `waitForStop`, `readRegisters`, `writeRegisters`, `readMemory`, `writeMemory`, `readMaps`, `readThreads`, `breakpoints`, `backtrace`, etc. Good news: the Windows core can implement the **same** signatures. |
| Types | `include/Types.h` | Uses `pid_t` (available in MinGW's `sys/types.h`). `MemoryRegion`, `ThreadInfo`, `RegisterSet`, `Breakpoint` are platform-neutral. |
| CLI | `include/HeadlessCLI.h`, `src/HeadlessCLI.cpp` (1,234 LoC) | Already headless / VT100-only. Only `<termios.h>` and `<unistd.h>` are POSIX-specific (used in `readLine`; both are provided by MinGW). |
| Platform adapters | `include/PlatformAdapters.h`, `src/PlatformAdapters.cpp` | Stubs only — `WindowsDebugAdapter` is documented as "Wine + gdbstub". The v1.2.0 plan replaces this with a real native Win32 path. |
| Stealth preload | `scripts/ninjastealth.c` (603 LoC) | Pure POSIX (`dlsym(RTLD_NEXT,...)`, `/proc/self/*` interception). Needs a separate Windows DLL implementation. |
| Build | `Makefile` | `g++` only. No cross-compile targets. `STEALTH_LIB = build/libninjastealth.so`. |
| MinGW on this host | `x86_64-w64-mingw32-g++` | **Not installed.** Must `apt install g++-mingw-w64-x86-64 mingw-w64-tools` (or dnf equivalent) before any Windows build can run. |

The plan below keeps the Linux ptrace implementation 100% intact and adds a parallel Windows implementation behind the same `DebuggerCore` interface, selected at compile time via `#ifdef _WIN32`. No refactor of existing Linux code is required.

### A.2 Implementation mapping — ptrace → Win32 Debug API

| NinjaDBG API (existing) | Linux ptrace impl (current) | Windows impl (new) |
|---|---|---|
| `attach(pid)` | `ptrace(PTRACE_ATTACH, pid)` + `waitpid` | `DebugActiveProcess(pid)` then loop `WaitForDebugEvent` until `CREATE_PROCESS_DEBUG_EVENT` arrives; cache `hProcess = OpenProcess(PROCESS_ALL_ACCESS, FALSE, pid)` |
| `attachByLaunch(path, args)` | `fork` → child `PTRACE_TRACEME` + `raise(SIGSTOP)` → `execv` | `CreateProcessA(NULL, cmdline, NULL, NULL, FALSE, DEBUG_ONLY_THIS_PROCESS, envp, cwd, &si, &pi)`; first debug event is `CREATE_PROCESS_DEBUG_EVENT` carrying `lpStartInfo`, `hProcess`, `hThread`, `lpBaseOfImage` |
| `detach()` | `PTRACE_DETACH` | `DebugActiveProcessStop(pid)` + `CloseHandle` on cached `hProcess`/`hThread`/module handles |
| `kill()` | `kill(pid, SIGKILL)` + `waitpid` | `TerminateProcess(hProcess, 1)` + `WaitForDebugEvent` until `EXIT_PROCESS_DEBUG_EVENT` |
| `cont()` | `PTRACE_CONT` | `ContinueDebugEvent(pid, tid, DBG_CONTINUE)` |
| `step()` | `PTRACE_SINGLESTEP` | `GetThreadContext` → set `EFLAGS |= 0x100` (TF) → `SetThreadContext` → `ContinueDebugEvent(..., DBG_CONTINUE)` → wait for `EXCEPTION_SINGLE_STEP` |
| `stop()` (pause) | `kill(pid, SIGSTOP)` | `DebugBreakProcess(hProcess)` |
| `waitForStop(sig, ex, code)` | `waitpid` + `WIFEXITED/WIFSTOPPED` macros | `WaitForDebugEvent(&de, INFINITE)` then a switch on `de.dwDebugEventCode`: `EXCEPTION_DEBUG_EVENT` (filter `EXCEPTION_BREAKPOINT`/`EXCEPTION_SINGLE_STEP` → "stopped by us"), `EXIT_PROCESS_DEBUG_EVENT` (set exited), `CREATE_THREAD_DEBUG_EVENT`, `EXIT_THREAD_DEBUG_EVENT`, `LOAD_DLL_DEBUG_EVENT`, `OUTPUT_DEBUG_STRING_EVENT` (swallow unless trace mode), `RIP_EVENT` |
| `readMemory(addr, buf, n)` | `process_vm_readv` (fallback `PTRACE_PEEKDATA`) | `ReadProcessMemory(hProcess, addr, buf, n, &done)` |
| `writeMemory(addr, buf, n)` | `process_vm_writev` (fallback `PTRACE_POKEDATA` with peek-overlay) | `WriteProcessMemory(hProcess, addr, buf, n, &done)`; if writing to .text, also `FlushInstructionCache(hProcess, addr, n)` |
| `readRegisters(out)` | `PTRACE_GETREGS` (fills `user_regs_struct`) | `Wow64GetThreadContext`/`GetThreadContext(hThread, &ctx)` with `ctx.ContextFlags = CONTEXT_FULL \| CONTEXT_DEBUG_REGISTERS`; map `ctx.Rax..R15` into `RegisterSet`; also populate DR0–DR3 + DR7 for watchpoints |
| `writeRegisters(in)` | `PTRACE_SETREGS` | Same in reverse: `SetThreadContext(hThread, &ctx)` |
| `addBreakpoint(addr, hardware=...)` | `setSwBreakpoint` writes 0xCC; HW bp TODO (currently degrades to SW) | SW: `ReadProcessMemory` 1 byte → save → `WriteProcessMemory` 0xCC; HW: pick first free DR slot (0..3) in DR7, write `ctx.Dr0..Dr3 = addr`, set L0..L3 + R/W + LEN bits in DR7, `SetThreadContext` |
| `readMaps()` | parse `/proc/<pid>/maps` | loop `VirtualQueryEx(hProcess, addr, &mbi, sizeof(mbi))` until `addr == 0`; map `mbi.State`/`mbi.Protect`/`mbi.Type` into existing `MemoryRegion.{read,write,exec,shared,path,offset}`. For `path`: use `GetModuleFileNameEx` if `mbi.Type == MEM_IMAGE` |
| `readThreads()` | enumerate `/proc/<pid>/task/` | `CreateToolhelp32Snapshot(TH32CS_SNAPTHREAD, pid)` + `Thread32First/Next`, filter `te.th32OwnerProcessID == pid` |
| `listProcesses()` (static) | parse `/proc/` | `CreateToolhelp32Snapshot(TH32CS_SNAPPROCESS, 0)` + `Process32First/Next`; map `pe32.szExeFile` and `pe32.th32ProcessID` |
| `backtrace()` | RBP chain walk via `readMemory` | Same RBP-chain walk works on Win64 if `Frame = RBP+0` holds the saved RBP. For PDB-symbolic names: optional `SymInitialize`/`StackWalk64` from `dbghelp.dll` (nice-to-have) |
| `currentSyscall()` | `orig_rax` from `user_regs_struct` | No direct equivalent (Win32 syscalls are not numbered via a syscall ABI). Return `-1` with a one-line warning, or stub it out via `ntdll!Nt*` instrumentation. Mark as Linux-only in CLI help. |
| `stepToSyscall` | `PTRACE_SYSCALL` | Not portable. Mark as Linux-only. |

### A.3 Multi-threaded event handling

Linux's `ptrace` follows clone/fork automatically (we set `PTRACE_O_TRACECLONE | TRACEFORK | TRACEEXEC | TRACEEXIT | EXITKILL`). On Windows the debug API receives `CREATE_THREAD_DEBUG_EVENT` and `EXIT_THREAD_DEBUG_EVENT` automatically — no opt-in needed. The Windows core must keep a `std::unordered_map<DWORD tid, HANDLE hThread>` updated as events arrive, so `step()`/`readRegisters()` can target the currently-stopped thread. Add `DebuggerCore::currentTid()` to the public API.

### A.4 Cross-compile with MinGW

**Toolchain install (Debian/Ubuntu):**
```
sudo apt install g++-mingw-w64-x86-64 mingw-w64-tools
# Provides: x86_64-w64-mingw32-g++ , x86_64-w64-mingw32-gcc , x86_64-w64-mingw32-windres
```
Fedora: `sudo dnf install mingw64-gcc-c++ mingw64-binutils`

**Compiler/Linker flags for the Windows build:**
```makefile
WIN_CXX      := x86_64-w64-mingw32-g++
WIN_CXXFLAGS := -std=c++17 -O2 -Wall -Wextra \
                -Iinclude -Isrc -D_WIN32_WINNT=0x0601 -DWINVER=0x0601 \
                -static -static-libgcc -static-libstdc++ -lwinpthread
WIN_LDFLAGS  := -lpsapi -ldbghelp -lntdll -lkernel32 -luser32
WIN_OBJ_DIR  := build/win/obj
WIN_BIN      := build/win/ninjadb.exe
WIN_DLL      := build/win/libninjastealth.dll
```
`-D_WIN32_WINNT=0x0601` targets Windows 7+ (required for `DebugSetProcessKillOnExit`, `Wow64GetThreadContext`). Static linking (`-static`) makes the produced `ninjadb.exe` portable: drop it on a clean Windows box and it runs without a MinGW runtime install.

**`#ifdef _WIN32` selection in `DebuggerCore.h`:**

Keep the existing class signature unchanged. Split the implementation:

```cpp
// DebuggerCore.cpp — keep as the Linux build (only compiled when ! _WIN32)
// DebuggerCoreWin.cpp — new file, only compiled when _WIN32
```
Both implement the **same** `DebuggerCore` class methods. The Makefile picks the right `.cpp` per target:

```makefile
ifeq ($(OS),Windows_NT)
    DBG_SRC := src/DebuggerCoreWin.cpp src/AntiDetectWin.cpp
else
    DBG_SRC := src/DebuggerCore.cpp src/AntiDetect.cpp
endif
```

For the MinGW cross-compile, force `DBG_SRC` to the Win variant regardless of host OS.

### A.5 New Makefile targets

Add to the existing `Makefile` (do not modify the existing `all`, `install`, `deb`, `clean`):

```makefile
# ---- Windows cross-compile (MinGW) -----------------------------------------
WIN_SRCS := src/DebuggerCoreWin.cpp src/AntiDetectWin.cpp \
            $(filter-out src/DebuggerCore.cpp src/AntiDetect.cpp, $(SRCS))
WIN_OBJS := $(patsubst src/%.cpp,$(WIN_OBJ_DIR)/%.o,$(WIN_SRCS))

.PHONY: win winexe windll wininstall winclean

win: winexe windll

$(WIN_OBJ_DIR)/%.o: src/%.cpp
	@mkdir -p $(dir $@)
	$(WIN_CXX) $(WIN_CXXFLAGS) -c $< -o $@

winexe: $(WIN_BIN)

$(WIN_BIN): $(WIN_OBJS)
	@mkdir -p $(dir $@)
	$(WIN_CXX) $(WIN_CXXFLAGS) $(WIN_OBJS) -o $@ $(WIN_LDFLAGS)
	@echo "[win] ninjadb.exe ready -> $@"

windll: $(WIN_DLL)

# libninjastealth.dll — MinHook-based DLL injection payload
$(WIN_DLL): scripts/ninjastealth_win.c scripts/MinHook.c scripts/MinHook.h
	@mkdir -p $(dir $@)
	x86_64-w64-mingw32-gcc -shared -O2 -Wall \
	    -D_WIN32_WINNT=0x0601 -DMINGW_HAS_SECURE_API=1 \
	    -o $@ scripts/ninjastealth_win.c scripts/MinHook.c \
	    -lntdll -lkernel32 -luser32 -lpsapi
	@echo "[win] libninjastealth.dll ready -> $@"

wininstall: win
	install -d $(DESTDIR)$(PREFIX)/bin
	install -m 0755 $(WIN_BIN) $(DESTDIR)$(PREFIX)/bin/ninjadb.exe
	install -m 0755 $(WIN_DLL) $(DESTDIR)$(PREFIX)/bin/libninjastealth.dll

winclean:
	rm -rf build/win/
```

### A.6 `libninjastealth.dll` — Windows hook list

Linux's `libninjastealth.so` is injected via `LD_PRELOAD` and uses `dlsym(RTLD_NEXT,...)`. On Windows the equivalent is **DLL injection** (CreateRemoteThread → LoadLibrary) plus an inline-hooking library. MinHook (3-clause BSD, MinGW-friendly) is the recommended hook engine.

**File:** `scripts/ninjastealth_win.c` + `scripts/MinHook.{c,h}` (vendored; ~2k LoC, public license).

**DLL entry-point:** `BOOL APIENTRY DllMain(HMODULE, DWORD reason, LPVOID)` — on `DLL_PROCESS_ATTACH`, initialize MinHook and apply all inline hooks. On `DLL_PROCESS_DETACH`, unhook everything.

**Hooks to install** (ordered by how commonly a packer/protector probes them):

| # | API | Module | Hook behavior | Defeats |
|---|---|---|---|---|
| 1 | `IsDebuggerPresent` | kernel32 | always return `FALSE` | Most basic check |
| 2 | `CheckRemoteDebuggerPresent` | kernel32 | set `*pbPresent = FALSE`; return `TRUE` | Common check |
| 3 | `NtQueryInformationProcess` | ntdll | intercept `ProcessDebugPort` (0x07) → return 0; `ProcessDebugFlags` (0x1F) → return 1; `ProcessDebugObjectHandle` (0x1E) → set handle=NULL, return `STATUS_PORT_NOT_SET` (0xC0000035) | The most common modern anti-debug |
| 4 | `NtSetInformationThread` | ntdll | swallow `ThreadHideFromDebugger` (0x11) — return `STATUS_SUCCESS` without calling real | Packers that try to hide threads |
| 5 | `OutputDebugStringA` / `OutputDebugStringW` | kernel32 | if running under a debugger, `GetLastError()` after the call returns 0; under no debugger it returns a specific value. We normalize: call real, then `SetLastError(0)` | OutputDebugString trick |
| 6 | `GetTickCount` / `GetTickCount64` | kernel32 | normalize across long calls (compute a per-call jitter delta and subtract) — optional, off by default | Simple timing checks |
| 7 | `QueryPerformanceCounter` | kernel32 | same idea as #6; off by default | High-res timing checks |
| 8 | `NtClose` | ntdll | if `hObject` is an invalid non-zero handle, swallow the `STATUS_INVALID_HANDLE` exception packers rely on | NtClose invalid-handle trap |
| 9 | `FindWindowA` / `FindWindowW` | user32 | if window-class name matches `OLLYDBG`, `WinDbgFrameClass`, `x64dbg`, `Qt5152QWindowIcon`, `ID` (Immunity), etc. → return `NULL` | Window-class scans |
| 10 | `BlockInput` | user32 | no-op (return `TRUE` without blocking) | Packers that block keyboard input |
| 11 | `CreateFileW` (optional) | kernel32 | if filename is `\\.\NtGlobalFlag` or starts with `\\.\PIPE\` for known emulators → return `INVALID_HANDLE_VALUE` | Sandbox/VM detection |
| 12 | `NtYieldExecution` / `SwitchToThread` | ntdll/kernel32 | call real but normalize timing — optional | Indirect timing checks |

**Configuration:** Each hook is toggled by an env var `NINJA_STEALTH_FLAGS` (bitmask). Default = bits 1+2+3+4+5+8+9+10 enabled; bits 6+7+11+12 (timing/`CreateFile` rewrite) disabled to avoid breaking legitimate programs.

**Injection:** `ninjadb.exe` injects the DLL into the target with the classic `CreateRemoteThread(hProcess, NULL, 0, (LPTHREAD_START_ROUTINE)LoadLibraryA, remotePath, 0, NULL)`. The `remotePath` is the absolute path to `libninjastealth.dll`, allocated in the target via `VirtualAllocEx` + `WriteProcessMemory`.

**Build:** MinGW cross-compile (see A.5). Output: `build/win/libninjastealth.dll` (~40–80 KB).

## PART B — x64dbg-like features to add

x64dbg's UI surfaces functionality that NinjaDBG's CLI lacks. Below: each feature → proposed CLI command → implementation requirements → priority. Priorities:

- **MUST** = ship-blocker for v1.2.0
- **SHOULD** = strongly desired for v1.2.0, can slip a week if needed
- **NICE** = future, post-v1.2.0

### B.1 Modules list — **MUST**

x64dbg's "Symbols" tab.

- **CLI:** `modules` (alias: `info modules`, `mod`)
- **Output columns:** `BASE               SIZE              ENTRY             NAME           PATH`
- **Implementation:**
  - Linux: parse `readMaps()` output, group by unique `path` where `offset == 0`, take the lowest `start` as base; for entry-point, read the ELF header at base (`e_entry` field).
  - Windows: `EnumProcessModulesEx(hProcess, hMods, sizeof(hMods), &cbNeeded, LIST_MODULES_ALL)` + `GetModuleInformation` (base/size) + `GetModuleBaseName`/`GetModuleFileNameEx`.
  - Add `DebuggerCore::readModules()` returning a new `std::vector<ModuleInfo>` (define `ModuleInfo { addr_t base; u64 size; addr_t entry; std::string name; std::string path; }` in `Types.h`).
- **Files touched:** `Types.h`, `DebuggerCore.h`, `DebuggerCore.cpp`, `DebuggerCoreWin.cpp`, `HeadlessCLI.{h,cpp}`.

### B.2 Handles list — **SHOULD**

x64dbg's "Handles" tab.

- **CLI:** `handles` (alias: `info handles`, `hdl`)
- **Output columns:** `HANDLE    TYPE         ACCESS  NAME`
- **Implementation:**
  - Linux: enumerate `/proc/<pid>/fd/` (each entry is a symlink — `readlink()` to get the target). Read `/proc/<pid>/fdinfo/<n>` for `flags` (O_RDONLY/O_WRONLY/O_RDWR). Also `/proc/<pid>/map_files/` for memory-mapped handles.
  - Windows: `NtQuerySystemInformation(SystemExtendedHandleInformation, ...)` filters by `UniqueProcessId == pid`; then `NtQueryObject(ObjectTypeInformation, ...)` and `NtQueryObject(ObjectNameInformation, ...)` for type/name. Requires `PROCESS_DUP_HANDLE` access.
  - Define `HandleInfo { u64 handle; std::string type; u32 access; std::string name; }` in `Types.h`.
- **Why not MUST:** Linux implementation is trivial (`/proc/<pid>/fd`), Windows implementation needs undocumented NtQuerySystemInformation — risky for v1.2.0 ship date. Ship the Linux half in v1.2.0; ship Windows half in v1.1.5.

### B.3 Memory dump to file — **MUST** (trivial)

x64dbg's "Save to file" in the dump pane.

- **CLI:** `dump <addr> <size> <file>` (e.g. `dump 0x401000 0x1000 out.bin`)
- **Implementation:**
  - `readMemoryVec(addr, size)` → `std::ofstream(file, std::ios::binary).write(...)`.
  - Add a sanity check: refuse to dump more than 256 MB without a `--force` flag.
  - Optional: `dump region <addr> <file>` — dumps the entire containing region from `readMaps()`.
- **Files touched:** `HeadlessCLI.{h,cpp}` only.

### B.4 Memory search — pattern (with wildcards) — **MUST**

x64dbg's "Find Pattern" (`Ctrl+B`).

- **CLI:** `find <start> <end> <pattern>` where `pattern` is space- or `\x`-separated hex bytes with `??` for wildcards. Example: `find 0x400000 0x410000 48 89 ?? ?? ?? ?? E8`
- **Output:** one line per match: `0x401234:  48 89 05 78 56 34 12 E8 ...`
- **Implementation:**
  - Parse pattern into `std::vector<std::optional<u8>>`.
  - Iterate the `[start, end)` range **page by page** (4 KB chunks via `readMemoryVec`), sliding a window over the buffer. For wildcard positions, skip the byte compare.
  - Optimization: Boyer-Moore-Horspool on the non-wildcard prefix; falls back to naive scan when too many wildcards.
  - Stop at first 1000 matches (configurable: `find -n 5000 ...`).
- **Files touched:** new `src/MemorySearch.cpp` + `include/MemorySearch.h`; CLI handler in `HeadlessCLI.cpp`.

### B.5 Memory search — string — **MUST**

x64dbg's "Find references to → String".

- **CLI:**
  - `findstr ascii <start> <end> <string>` — ASCII / UTF-8
  - `findstr utf16 <start> <end> <string>` — UTF-16LE
  - `findstr all <start> <end> [minlen]` — enumerate all printable strings ≥ `minlen` chars in range
- **Implementation:** Reuses B.4's chunked reader. For `ascii`/`utf16` sub-modes: build the search pattern from the input string and feed it to B.4's matcher. For `all` mode: standard string-extraction algorithm (run of `0x20..0x7E` for ASCII, alternating `0x20..0x7E` / `0x00` for UTF-16).
- **Files touched:** extends `MemorySearch.{h,cpp}`; CLI handler.

### B.6 Run trace (instruction tracing) — **MUST**

x64dbg's "Run Trace" — records every executed instruction's RIP, registers, and (optionally) memory accesses.

- **CLI:**
  - `trace on  [count]` — start tracing for the next `count` single-steps (default: unlimited until breakpoint/exit)
  - `trace off` — stop tracing
  - `trace save <file>` — write the captured trace to a file (one instruction per line)
  - `trace status` — show how many instructions captured so far
- **Trace line format:** `#00001234  RIP=000000004010F2  48 89 E5              mov rbp, rsp   | RAX=0xABCD RBX=... (only deltas after first)`
- **Implementation:**
  - New `class RunTrace` in `include/RunTrace.h` with `void record(DebuggerCore&)`, `void save(const std::string& path)`, `void clear()`, `size_t count()`.
  - In trace mode, the CLI's `step`/`cont` handlers single-step in a loop, calling `RunTrace::record()` after each step. After each step: read registers, disassemble one instruction at RIP (reuse the existing `Disassembler`), compute register delta vs. the previous step, append to an in-memory `std::vector<std::string>` (cap at 10M entries; warn if exceeded).
  - To trace a running target without typing `step` repeatedly, add `trace run [count]` which loops: `dbg_.step()` → `record()` → check for exit/breakpoint → repeat.
- **Files touched:** new `RunTrace.{h,cpp}`; CLI handlers.

### B.7 CPU flags display — **MUST** (trivial)

x64dbg shows each RFLAGS bit by name. NinjaDBG currently prints `RFL = 0x...`.

- **CLI:** `info flags` (alias: `flags`)
- **Also:** include in `info registers` output as a second block.
- **Output:**
  ```
  CF=0 PF=1 AF=0 ZF=1 SF=0 TF=0 IF=1 DF=0 OF=0
  IOPL=0  NT=0  RF=1  VM=0  AC=0  VIF=0  VIP=0  ID=1
  ```
- **Implementation:** Pure bit manipulation on `RegisterSet::rflags`:
  - CF = bit 0, PF = bit 2, AF = bit 4, ZF = bit 6, SF = bit 7, TF = bit 8, IF = bit 9, DF = bit 10, OF = bit 11, IOPL = bits 12-13, NT = bit 14, RF = bit 16, VM = bit 17, AC = bit 18, VIF = bit 19, VIP = bit 20, ID = bit 21.
- **Files touched:** `HeadlessCLI.cpp::printRegisters()`; add `cmdInfo` branch for `"flags"`.

### B.8 Register modification — **MUST** (trivial)

x64dbg lets you double-click any register and type a new value.

- **CLI:** `set reg <name> = <value>`  (e.g. `set reg rax = 0xDEADBEEF`)
- Also support: `set reg rip = 0x401000` (jump), `set reg rflags = 0x246` (full rewrite), `set reg zf = 1` / `set reg cf = 0` (single-flag toggles).
- **Implementation:**
  - Parse register name (case-insensitive) → look up in `kRegNames` table → index into `RegisterSet` field.
  - For single-flag setters (`cf`, `pf`, `zf`, ...): read registers, set/clear the bit (mapping same as B.7), write back.
  - Reject writes when not attached; warn if writing `rip` while target is mid-instruction (RIP must point to a valid instruction boundary).
- **Files touched:** `HeadlessCLI.cpp::cmdSet()` (extend existing `set` command).

### B.9 Detailed memory map — **MUST**

x64dbg's "Memory Map" tab shows far more than NinjaDBG's current `info maps`.

- **CLI:** `info map` (alias: `map`)
- **New output columns:** `BASE               SIZE               STATE   TYPE       PROTECT          RSS  PATH`
  - `STATE`: commit / reserve / free (Windows) or "backed" / "anon" (Linux via `/proc/<pid>/smaps`)
  - `TYPE`: image / mapped / private (Windows) or file / anon / heap / stack (Linux)
  - `PROTECT`: rwxp-style string (already present) but also PAGE_EXECUTE_READWRITE style for Windows
  - `RSS`: resident set size in KB (Linux only — from `/proc/<pid>/smaps` field `Rss:`)
- **Implementation:**
  - Extend `MemoryRegion` struct in `Types.h`:
    ```cpp
    enum class RegionState { Commit, Reserve, Free, Backed, Anonymous };
    enum class RegionType  { Image, Mapped, Private, File, Anon, Heap, Stack };
    RegionState state = RegionState::Commit;
    RegionType  type  = RegionType::Private;
    std::string protect_str;          // human-readable "RWX"
    u64         rss_kb = 0;           // Linux only
    u64         private_kb = 0;       // Linux only (Pss from smaps)
    ```
  - Linux: parse `/proc/<pid>/maps` as today, then for each region look up the matching `/proc/<pid>/smaps` block to fill `rss_kb` and `private_kb`.
  - Windows: `VirtualQueryEx` already returns `STATE`/`TYPE`/`PROTECT` directly; map to the enum.
  - Keep the old `info maps` command as an alias for backward compatibility.
- **Files touched:** `Types.h`, `DebuggerCore.{h,cpp}`, `DebuggerCoreWin.cpp`, `HeadlessCLI.cpp::printMaps()`.

## D. Recommended implementation order

1. **Day 1–2 — Types + Linux-only features (B.3, B.4, B.5, B.7, B.8)**. These touch only `HeadlessCLI.cpp` + new files; no platform work. Ship-able in the Linux build immediately, validated with `tests/test_basic`.
2. **Day 3 — B.9 detailed memory map (Linux side)**. Extends `MemoryRegion` + `readMaps` to also read `/proc/<pid>/smaps`.
3. **Day 4 — B.1 modules (Linux side)**. Group `readMaps` results by path.
4. **Day 5 — B.6 run trace**. Pure addition; uses existing `Disassembler` and `DebuggerCore::step`.
5. **Day 6–8 — Windows core** (`DebuggerCoreWin.cpp`, `AntiDetectWin.cpp`, `WinApiShim.h`). Cross-compile with MinGW; smoke-test on a Windows VM using a trivial test program.
6. **Day 9–10 — `libninjastealth.dll`**. Vendor MinHook, write the 12 hooks, test injection via `CreateRemoteThread`.
7. **Day 11 — B.1/B.9 Windows side**. `EnumProcessModulesEx` + `VirtualQueryEx` plumbing.
8. **Day 12 — B.2 handles (Linux only)**. `/proc/<pid>/fd` + `/proc/<pid>/fdinfo`.
9. **Day 13 — Makefile + README + INSTALL**.
10. **Day 14 — Tag v1.2.0, build .deb + Windows .zip artifacts.**

## E. Risks & open questions

- **MinGW availability on the dev box.** The current sandbox does **not** have `x86_64-w64-mingw32-g++` installed. Step 0 of any implementation is `sudo apt install g++-mingw-w64-x86-64 mingw-w64-tools`. Until that's done, only the Linux-side features (B.1, B.3–B.9) can actually be built and tested.
- **MinHook licensing.** MinHook is 3-clause BSD — compatible with NinjaDBG's Apache-2.0. Vendor it under `scripts/MinHook.{c,h}` and add a `THIRD_PARTY.md` entry. Alternatively, write a 200-LoC inline-hook helper ourselves to avoid the dependency; recommend vendoring MinHook to save time.
- **`ptrace` semantics don't map 1:1 to Win32.** Specifically: (a) `stepToSyscall` has no Win32 equivalent — document as Linux-only; (b) hardware watchpoints via DR0–DR3 on Windows require setting `CONTEXT_DEBUG_REGISTERS` and re-applying after every context switch; the implementation must re-assert DR7 on every `WaitForDebugEvent` wakeup.
- **DLL injection via `CreateRemoteThread + LoadLibrary` is detected by some packers.** A more stealthy injector (`NtCreateThreadEx`, manual map) is out of scope for v1.2.0 — note it in the README as a known limitation; the v1.1.5 plan addresses it.
- **`process_vm_readv` on Linux and `ReadProcessMemory` on Windows both fail on unmapped pages.** The new memory-search and dump commands must catch these errors per-page and skip, not abort the whole operation.
- **EULA / license headers.** Every new `.cpp`/`.h`/`.c` file must start with the same `// NinjaDBG v1.2.0 - <Module> - Open Source (Apache-2.0) - by Chapzoo` header used elsewhere.
